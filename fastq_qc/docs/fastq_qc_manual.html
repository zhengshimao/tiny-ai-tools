fastq_qc

Lightweight FASTQ Quality Control Tool — 提取并改写自 fastp v1.3.3 的核心质控算法。 支持单端/双端 FASTQ，自动识别 gzip 压缩，输出文本摘要、JSON 报告、TSV 表格。

1. 概述与设计思路

fastq_qc 从 fastp 中提取了质量指标统计引擎，重写为独立的单文件 C++ 工具。设计目标：

  • 轻量 —— 单文件源码，仅依赖 zlib
  • 兼容 fastp —— 核心数据布局（34 数组循环统计）与 fastp 一致
  • 双端感知 —— 单端/双端共用同一代码路径，自动适配
  • 多格式输出 —— 文本摘要、JSON 报告、TSV 表格任选组合
  • gzip 透明读取 —— 通过扩展名和 magic bytes 自动识别

2. 编译安装

$ cd fastq_qc $ make g++ -O3 -Wall -Wextra -std=c++0x -o fastq_qc fastq_qc.cpp -lz $ ./fastq_qc -h Usage: ./fastq_qc [options] -i <read1.fastq> [-I <read2.fastq>] ...
依赖：g++ ≥ 4.6，zlib (libz-dev / zlib-devel)。 Windows 下建议在 WSL 中编译。

3. 使用示例

3.1 单端 FASTQ，仅文本摘要

$ ./fastq_qc -i sample.fq.gz

3.2 单端 FASTQ + JSON 报告

$ ./fastq_qc -i sample.fq.gz -o qc_report.json

3.3 双端 FASTQ + JSON + TSV

$ ./fastq_qc -i R1.fq.gz -I R2.fq.gz -o report.json -p sample1

3.4 百分比模式 + 千分位 + 样本 ID

$ ./fastq_qc -i R1.fq.gz -I R2.fq.gz \ -o report.json -p sample1 -s healthy12 \ -percent -d 2 --tsv-comma

3.5 Phred+64 旧格式数据

$ ./fastq_qc -i old_illumina.fastq -6 -o qc_report.json

3.6 从 stdin 读取

$ cat sample.fastq | ./fastq_qc -i -

提示：以上示例中反斜杠 \ 是 bash 的换行续行符，用于将长命令拆分为多行，实际输入时可写为一行或保留 \ 换行。

6. 类与函数说明

6.1 FastqRead struct

存储一条 FASTQ 记录的四个字段：

字段类型说明
namestring标识行 (以 @ 开头)
seqstring碱基序列
strandstring链标识行 (以 + 开头)
qualitystringPhred+33 质量字符串

6.2 FastqReader class

负责从文件（或 stdin）读取 FASTQ 记录。使用 4MB 内部缓冲区减少系统调用，类似 fastp 的环形缓冲区设计。

方法说明
FastqReader(filename)构造：检测 gzip (扩展名 + magic bytes)，打开文件
read(FastqRead&)读取一条记录，返回 false 表示 EOF
has_gz_ext()检查 .gz / .GZ 扩展名
has_gz_magic()检查 gzip magic bytes (0x1F 0x8B)
read_chunk()从下层文件填充内部缓冲区
next_line(string&)提取下一行（处理跨缓冲区边界）

6.3 QcStats class

核心质量统计引擎，数据布局与 fastp 的 Stats 类一致：34 个 long 数组（Q30[8] + Q20[8] + Content[8] + Qual[8] + TotalBase + TotalQual）× bufLen。

方法说明
process_read(r)处理一条 read，更新 per-cycle 计数和直方图
summarize()汇总全局指标、生成质量/含量/GC 曲线
merge(other)合并另一个 QcStats（用于双端合并）
print()输出对齐的文本摘要（stdout）
report_json(os)输出 JSON 片段
write_cycles_tsv(os)输出 per-cycle TSV 表
gc_content()GC 含量（受 -percent 控制）
q20/30/40_rate()Q20/Q30/Q40 比例
cycle20_rate()第 20 个测序循环 (0-based 索引 19) 处质量 ≥ Q20 的碱基比例
mean_length()平均 read 长度
length_histogram()read 长度分布直方图 (map<int,long>)，每处理一条 read 自动更新
write_length_tsv(os)输出 read 长度分布 TSV (两列: length, count)
write_length_percent_tsv(os)输出 read 长度百分比 TSV (两列: length, percent)

6.3.1 碱基编码 (base2idx)

fastp 利用 ASCII % 8 的性质快速映射碱基：

碱基ASCIIASCII % 8
A651
T844
C673
G717
N786
static inline int base2idx(char base) { return base & 0x07; }

6.4 DupCounter class

基于 FNV-1a 64-bit hash + BloomFilter（固定 256 MB）的去重统计，线程安全（mutex 保护）。双端时将 R1 和 R2 的 hash 通过 boost::hash_combine 合并。

方法说明
add_read(r)单端：hash 序列并判重
add_pair(r1, r2)双端：合并 R1+R2 hash 判重
dup_rate()返回原始比例 (0.0–1.0)

6.5 InsertSizeStats class

仅双端生效。将 R2 反向互补后，与 R1 后缀做容错重叠匹配（≤5 mismatch 或 20%），计算 insert_size = len(R1) + len(R2) - overlap。

方法说明
process_pair(r1, r2)计算 insert size 并更新直方图
peak()直方图众数（peak insert size）
unknown()无法确定 insert size 的 pair 数

7. 算法详解

7.1 质量统计 (QcStats)

Per-Cycle 34 数组布局

单次分配 34 × bufLen 个 long，由 QcStats 内部管理指针：

m_cycle_buf = new long[34 * bufLen];

// 数组布局:
m_cycle_q30[0..7]      = m_cycle_buf + [ 0..7]  * bufLen  // 每种碱基的 Q30 次数
m_cycle_q20[0..7]      = m_cycle_buf + [ 8..15] * bufLen  // 每种碱基的 Q20 次数
m_cycle_content[0..7]  = m_cycle_buf + [16..23] * bufLen  // 每种碱基出现次数
m_cycle_qual[0..7]     = m_cycle_buf + [24..31] * bufLen  // 每种碱基质量值总和
m_cycle_total_base     = m_cycle_buf + [32]     * bufLen  // 每 cycle 总碱基数
m_cycle_total_qual     = m_cycle_buf + [33]     * bufLen  // 每 cycle 总质量值

动态扩容

当 read 长度超过当前 bufLen 时，extend_buffer(new_len) 分配新数组，逐行 memcpy 旧数据，确保不丢已有统计。

汇总 (summarize)

遍历所有 cycle，计算：

  • 全局指标：total_bases, Q20/Q30/Q40 totals, GC bases, mean_length
  • 质量曲线：mean_qual, A/T/C/G per-base mean qual
  • 含量曲线：A/T/C/G/N per-cycle fraction, GC per-cycle fraction
  • 长度分布：每条 read 处理时自动统计到 m_length_hist 直方图中，汇总后可输出 count 和 percent 两种 TSV

7.2 重复检测 (DupCounter)

读取序列
↓
FNV-1a 64-bit hash
↓
hash 在 BloomFilter 中（可能重复）
hash 不在 BloomFilter 中（首次出现）
→ m_dup++ → 标记 bit 位
↓
dup_rate = m_dup / m_total
双端处理：R1 和 R2 的 hash 通过 boost::hash_combine 合并为一个 64-bit 值再判重。

7.3 插入片段大小 (InsertSizeStats)

R2 反向互补
↓
从最大可能重叠开始 → 递减至 30bp
↓
容错匹配（≤5 mismatch）
无匹配 ≥ 30bp
→ isize = len1 + len2 − overlap → m_unknown++
↓
更新直方图; peak = 众数

7.4 cycle20_rate 定义说明

"cycle20" 一词在不同上下文中含义不同，本工具的 cycle20_rate 特指：

第 20 个测序循环（即 reads 中第 20 个碱基位置，0-based 索引 19）处，质量值 ≥ Q20 (Phred 20, ASCII '5') 的碱基占比。

计算公式：cycle20_rate = m_cycle_q20[0..7][19] 之和 / m_cycle_total_base[19]，结果受 -percent / -d 选项控制。

常见歧义对照：
上下文cycle20 含义
本工具 fastq_qc位置 20 处 Q20+ 碱基比例 (cycle20_rate)
某些 QC 报告Q20 对应的 Phred 质量阈值 (ASCII '5' / Phred 20)
某些 QC 报告 / 本工具早期版本平均质量首次降至 Q20 以下的 cycle 编号 (cycle Q20 drop position)

如需全局 Q20 比例，请使用 q20_rate 指标；如需 Q20 阈值本身，参考常量定义 (Phred+33 编码下 ASCII='5')。

11. 注意事项

cycle20 定义："cycle20" 一词在不同报告中含义不同，本脚本中的 cycle20_rate 特指第 20 个测序循环（即 reads 中第 20 个碱基位置，0-based 索引 19）处，质量值 ≥ Q20 的碱基占比。

注意区分以下常见歧义：
  • 本脚本：cycle20_rate = 位置 20 处的 Q20+ 比率
  • 某些报告：cycle20 = Q20 对应的 Phred 阈值（即 ASCII '5' / Phred 20）
  • 某些报告：cycle20 = 平均质量首次降至 Q20 以下的 cycle 编号（本脚本早期版本曾如此定义，已改为当前定义）
如果您期望的是其他语义，请注意区分，或参考本工具的其他指标（如 q20_rate 为全局 Q20+ 比例）。
内存使用：DupCounter 使用固定 256 MB 的 BloomFilter，内存不随数据量增长。
插入片段大小：仅当 -I 指定 R2 时计算。估算依赖重叠分析（≥30bp 精确匹配），不重叠的 pair 归入 unknown。
Q40：Q40 基数为 0 是正常的——多数 Illumina 数据的质量上限为 Q38–Q42，Q40 阈值 (ASCII 'I' = 73) 通常只有极少数碱基达到。
单端重复率：单端模式下的重复率可能高于双端——因为仅用一条 read 的 hash，碰撞概率更高。这与 fastp 的行为一致。
TSV 千分位：开启 --tsv-comma 后数值列会包含逗号，部分解析工具可能需要预处理。默认不开启。
JSON 千分位：开启 --json-comma 后数值以字符串 "61,329" 输出，不再是 JSON number 类型。下游解析需注意转换。
Phred+64 编码：旧版 Illumina 数据（1.8 之前）使用 Phred+64 质量编码。使用 -6 或 --phred64 选项后，FastqReader 会在读取时将质量值自动转换为 Phred+33（每个碱基 ASCII 码减 31，下限为 33），后续所有分析均基于 Phred+33。
gzip 检测：通过文件扩展名 .gz 和 magic bytes 0x1F 0x8B 双重检测。即使文件没有 .gz 扩展名，只要内容以 gzip 魔数开头即可正确解压。
